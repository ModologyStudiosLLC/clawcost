The Hidden Costs of Running AI Agents in Production

Your cost estimate was based on what you could see: tokens in, tokens out. But production agents have seven additional cost sources that never appear in a prototype — and together they're often larger than the base token cost.

When developers estimate the cost of an AI agent, they typically model the happy path: a clean task, a few turns, straightforward tool calls. That model misses most of what actually drives production costs.

Here are the seven hidden cost sources, with real numbers for each.

1
Tool Call System Prompt Overhead

Every API call that includes tool definitions adds a hidden system prompt that explains to the model how tool use works. Anthropic's pricing documentation is explicit: using tools with tool_choice: auto or none adds 346 tokens to the system prompt. Using any or a specific tool adds 313 tokens.

That's before your actual tool definitions. A tool with a name, description, and a JSON schema for its parameters typically runs 100–400 tokens per tool. Define 5 tools and you're adding 500–2,000 tokens to every single call — including calls where the model doesn't use any tool.

Example: 10 tools × 200 tokens each + 346 overhead = 2,346 extra input tokens per call. At Claude Sonnet 4.6 ($3/MTok), that's $0.007 per call in pure overhead. At 50,000 calls/month: $350/month in invisible input costs.

Fix: Pass only the tools needed for the current step. In a multi-stage pipeline, provide a subset of tools per stage. Remove tools when the current step doesn't require tool use at all.

2
Retry Costs on Failed Tool Calls

When a tool call fails — network timeout, external API rate limit, malformed response — most agent frameworks retry the entire LLM call with an error message appended to the context. Each retry is billed at the same rate as a successful call, plus it grows the context for future turns.

A retry on turn 8 of a conversation with 12,000 tokens of accumulated context costs the same as the original turn 8 call. If your external tool has a 5% error rate and you allow 3 retries, that's 15% of your calls generating up to 3x the expected cost for that turn.

Example: Turn 8 context is 12,000 tokens in, 800 tokens out. Cost: $0.048. With 3 retries on failures (5% rate across 10,000 daily calls = 500 failures), that's 1,500 extra calls × $0.048 = $72/day in retry costs — roughly $2,160/month.

Fix: Handle tool errors at the tool wrapper layer, not the LLM layer. Return a structured error object to the model on first failure, not a full retry. Only escalate to a new LLM call if the error genuinely requires the model to decide what to do next. Set a maximum retry count and break the loop.

3
Growing Conversation History

In multi-turn agents, the full conversation history is sent on every call. This means the cost of turn N includes all tokens from turns 1 through N-1. For a 15-turn conversation where each turn adds 800 tokens, by turn 15 you're sending roughly 12,000 tokens of history on what started as a 500-token request.

Fix: Use prompt caching on the stable portions (system prompt, early conversation) and implement conversation summarization to compress older turns. See our post on prompt caching for the exact math.

4
Tool Result Tokens

When a tool returns data, that data becomes input tokens on the next call. A web search that returns 3 results at 500 words each adds roughly 2,000 tokens to your context. A database query that returns a JSON payload of 50 rows adds 3,000–8,000 tokens.

These tokens are invisible in most cost estimates because they're not part of your original prompt — they come from tool execution. But you pay for them at the full input rate.

Example: An agent calls a search tool 4 times per task, each returning ~1,500 tokens of results. That's 6,000 additional input tokens per task. At Claude Sonnet 4.6: $0.018 per task. At 10,000 tasks/month: $180/month in tool result tokens alone.

Fix: Truncate or summarize tool results before passing them to the model. A search result doesn't need to include the full article body — a 200-token excerpt and URL is usually sufficient. A database result doesn't need all 50 rows — return only the relevant ones. Use a lightweight model to summarize tool results before passing to the main agent model.

5
Embedding Costs

RAG pipelines generate embeddings for every document chunk at index time, and for every user query at search time. These are billed separately from completion tokens. OpenAI's text-embedding-3-large costs $0.13/MTok. Smaller models are cheaper but may reduce retrieval quality.

For a knowledge base of 100,000 document chunks averaging 500 tokens each: 100,000 × 500 = 50 million tokens to embed. At $0.13/MTok: $6.50 one-time indexing cost. Not huge, but each re-index (new documents, updated content) incurs it again.

More importantly, every user query generates an embedding. At 50,000 queries/day with 50-token average queries: 2.5 million tokens/day × $0.13/MTok = $0.33/day, $10/month. Easy to overlook in the primary model cost analysis.

Fix: Cache embeddings for common queries. Use a smaller embedding model if retrieval quality holds up. Batch your re-indexing operations using the Batch API (50% discount where available).

6
Vision and Multimodal Tokens

Images are tokenized at a rate that depends on image dimensions and the provider's tiling approach. Anthropic counts image tokens based on image size: a 1080×1080 image uses approximately 1,334 tokens (as a high-resolution image). A full-page PDF screenshot at 2x resolution can consume 4,000–6,000 tokens per page.

If your agent processes documents by screenshotting pages and passing them to a vision model, a 10-page document could cost $0.03–$0.09 in input tokens just for the images — before any text tokens from the prompt or OCR results. At scale this adds up fast.

Fix: Use text extraction (PDF parsing, OCR) instead of vision where possible — text tokens are much cheaper than image tokens. Reserve vision models for cases where the visual layout genuinely matters (charts, forms, diagrams).

7
Sub-agent Spawning

Multi-agent architectures — where an orchestrator spawns specialist sub-agents to handle parts of a task — multiply your cost by the number of agents running in parallel. If an orchestrator spawns 3 sub-agents, each with their own tool calls and conversation history, the total cost is the sum of all four agents' API calls, not just the orchestrator's.

This is often intentional and correct. But it means cost estimation requires modeling the entire call tree, not just the top-level agent. An orchestrator that makes 2 calls but spawns 3 sub-agents each making 8 calls has a true call count of 26, not 2.

Fix: Set per-agent token budgets and enforce them at the proxy level. Monitor cost per sub-agent type separately so you can see which specialist is driving spend. Consider whether parallel sub-agents are necessary — sequential execution is cheaper when latency requirements allow it.
