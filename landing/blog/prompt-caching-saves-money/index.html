Prompt Caching: Does It Actually Save Money?

Anthropic charges a premium to write content to cache — and then charges only 10% of normal input price on reads. We ran the break-even math on common workloads so you know exactly when caching helps, when it doesn't, and how much you can actually save.

Prompt caching sounds like an obvious win: store your system prompt once, read it cheaply forever. But the pricing model is more nuanced than that — you pay a write premium, and the benefit only materializes if the cache is hit enough times before it expires. Whether caching saves money depends entirely on your request volume and cache hit rate.

How Anthropic's Prompt Caching Pricing Works

Anthropic offers two cache durations, each with different write costs and the same read cost:

OperationPrice multiplierEffective price (Sonnet 4.6)Cache duration
Standard input (no cache) 1.0x $3.00 / MTok —
Cache write (5-minute TTL) 1.25x $3.75 / MTok 5 minutes
Cache write (1-hour TTL) 2.0x $6.00 / MTok 1 hour
Cache read (hit) 0.1x $0.30 / MTok —

The minimum cacheable block is 1,024 tokens. If your system prompt is under 1,024 tokens, you can't cache it at all — you have to pad it or accept standard pricing.

Prompt caching is available on Claude Sonnet 4.6, Claude Opus 4.6, Claude Haiku 4.5, and other current Claude models. The same multipliers apply regardless of model — only the base price changes.

The Break-Even Calculation

For caching to save money, the total cost of (1 write + N reads) must be less than the cost of (N+1 standard reads).

5-minute cache break-even
1.25x write + N × 0.1x read < (N + 1) × 1.0x standard
Solving for N: break-even at N = 1 read

For the 5-minute TTL cache, you break even after just one cache read. The write costs 1.25 units; one read costs 0.10 units; total: 1.35 units. Without caching, two standard calls cost 2.0 units. You're already ahead by the second request.

1-hour cache break-even
2.0x write + N × 0.1x read < (N + 1) × 1.0x standard
Solving for N: break-even at N = 2 reads

For the 1-hour TTL, you need two cache reads within an hour to break even. Write costs 2.0; two reads cost 0.20; total: 2.20 units. Without caching, three standard calls cost 3.0 units. You're ahead starting with the third request.

Real-World Savings Scenarios

Scenario 1: System Prompt in a Chatbot (High Volume)

Setup: 2,000-token system prompt, 1,000 users per day, 5 turns per user average. All turns share the same system prompt.

  • Without caching: 5,000 calls × 2,000 tokens = 10M input tokens/day at $3/MTok = $30/day
  • With 5-min caching (assuming 1 write + 4 reads per session, 1,000 sessions): 1,000 writes × 2,000 × $3.75/MTok + 4,000 reads × 2,000 × $0.30/MTok = $7.50 + $2.40 = $9.90/day
  • Savings: $20.10/day ($603/month) — 67% reduction on system prompt costs alone

Scenario 2: Large Document Analysis (Sporadic Requests)

Setup: 50,000-token document, users analyze it 3 times over a 30-minute window, then it's done.

  • Without caching: 3 calls × 50,000 tokens = 150,000 input tokens × $3/MTok = $0.45
  • With 5-min caching: 1 write × 50,000 × $3.75/MTok + 2 reads × 50,000 × $0.30/MTok = $0.1875 + $0.03 = $0.2175
  • Savings: $0.2325 per document — 52% reduction

Scenario 3: System Prompt in a Low-Volume Internal Tool

Setup: 2,000-token system prompt, 20 calls per day with gaps of 30+ minutes between calls.

  • Without caching: 20 calls × 2,000 tokens × $3/MTok = $0.12/day
  • With 5-min caching (cache expires between most calls, so many writes): 15 writes × 2,000 × $3.75/MTok + 5 reads × 2,000 × $0.30/MTok = $0.1125 + $0.003 = $0.1155/day
  • Savings: $0.0045/day — not worth the implementation complexity

The third scenario is the important one: caching only helps when there are multiple requests within the TTL window. For low-volume or infrequent workloads, you might actually pay more if you use the 1-hour TTL on a cache that never gets read.

When Prompt Caching Backfires

The 1-hour TTL trap

If you set a 1-hour cache on a 5,000-token document and only receive 1 request for it in that hour, you've paid 2.0x for the write and 0x for reads — costing you double the standard price. The longer TTL only makes sense for content that's accessed frequently within its window. For sporadic document analysis, the 5-minute TTL is almost always better: the write premium is lower, and you still capture multi-turn conversations.

How to Implement It in Practice

Anthropic supports two caching modes:

Automatic caching (recommended)

Add a single top-level cache_control flag and Anthropic manages breakpoints automatically as the conversation grows:

{
  "model": "claude-sonnet-4-6",
  "cache_control": {"type": "ephemeral"},
  "system": "Your system prompt here...",
  "messages": [...]
}

Explicit cache breakpoints (fine-grained control)

Place cache_control on specific content blocks when you want precise control — for example, caching a large document separately from the system prompt:

{
  "model": "claude-sonnet-4-6",
  "system": [
    {
      "type": "text",
      "text": "Your system prompt...",
      "cache_control": {"type": "ephemeral"}
    }
  ],
  "messages": [
    {
      "role": "user",
      "content": [
        {
          "type": "text",
          "text": "Here is the document to analyze:\n\n[50,000 token document]",
          "cache_control": {"type": "ephemeral"}
        },
        {
          "type": "text",
          "text": "What are the key findings?"
        }
      ]
    }
  ]
}

On subsequent turns in the same conversation, the system prompt and document are read from cache at 0.1x cost. Only the new user message and conversation extension are billed at full input price.

Stacking Caching with Batch API

Anthropic allows prompt caching and the Batch API discount to stack. If you're running batch jobs with repeated system prompts or documents:

ScenarioInput cost (Sonnet 4.6)Effective rate
Standard (no caching, no batch)$3.00 / MTokbaseline
Batch API only$1.50 / MTok50% off
Cache reads only$0.30 / MTok90% off
Cache reads + Batch API$0.15 / MTok95% off

For a large-scale document processing pipeline that repeats the same system prompt across thousands of documents sent as a batch job, the stacked discount brings your effective input cost to $0.15/MTok — 20x cheaper than standard pricing.
