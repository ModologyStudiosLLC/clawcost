Why Your LLM Costs Are 10x Higher Than Expected (And How to Fix It)

You looked at the pricing page, did the math, and felt confident about the budget. Then the first monthly bill arrived. Here are the six reasons that happen — and the specific fix for each one.

The mental model most developers use when estimating LLM costs goes something like: "I'll send about 1,000 tokens per request and get back 500 tokens. At $3/MTok input and $15/MTok output, that's $0.0105 per call. Easy."

That model is almost always wrong. Not slightly wrong — often off by 5–10x in production. Here's why.

Problem 1: Context Window Bloat

Every API call bills you for the entire context sent — system prompt, conversation history, retrieved documents, tool definitions, and the current user message. In multi-turn applications, the context grows with every exchange, and you pay for all of it on every call.

A typical pattern: turn 1 sends 500 tokens. Turn 5 sends 3,500 tokens (because the previous four turns are now in the context). Turn 10 sends 7,000 tokens. You estimated 500 per call; you're paying 10–14x that by mid-conversation.

Fix

Implement a sliding context window or conversation summarization. After N turns, summarize older turns into a compact summary and drop the raw history. You pay once for the summarization call, then send a condensed context going forward. Combined with prompt caching on the stable parts, this can cut context costs by 60–80%.

Problem 2: Tool Call Token Overhead

When you use tool use / function calling, you're paying for more tokens than you see. Anthropic's pricing docs document this explicitly: every API call with tools enabled adds a 346-token system prompt that the model needs to understand how tool use works. Every tool definition you pass also consumes input tokens.

A typical setup with 5 tools, each with a name, description, and JSON schema, can easily add 1,000–2,000 tokens to every single request — before the user's message is even counted. If you're running 100,000 agent calls per month with 5 tools defined, you might be paying for 100–200 million invisible input tokens.

At Claude Sonnet 4.6 ($3/MTok input), that's $300–$600/month in overhead you never see in your prompt.

Fix

Only pass tools the model actually needs for the current step. In agentic pipelines, provide a subset of tools per stage rather than the full toolset on every call. Trim tool descriptions to the minimum needed — a 30-word description is usually as useful as a 200-word one. Remove tools entirely when you're in a step that doesn't need them (tool overhead drops to zero with tool_choice: none).

Problem 3: Output Verbosity

Output tokens cost 4–5x more than input tokens. This is the most expensive line on your bill, and it's often controllable.

LLMs default to verbose, explanatory output — especially when given no instructions about length. A model asked to "check if this email is a phishing attempt" might return three paragraphs of reasoning when you only need a boolean and a confidence score. At $15/MTok for Claude Sonnet 4.6 output, a 1,000-token response costs $0.015. The same task answered in 50 tokens costs $0.00075 — a 20x difference.

Real example: classification endpoint

A developer running a sentiment classification endpoint was averaging 800 output tokens per call. After adding "Respond with only a JSON object containing 'label' and 'confidence'. No explanation." to the system prompt, average output dropped to 35 tokens. Monthly output token cost fell from ~$960 to ~$42 on 80,000 daily calls — same model, same accuracy.

Fix

Be explicit about output format and length in your system prompt. For structured outputs, use response_format: json_schema (OpenAI) or tool use with a schema (Anthropic) to constrain the response shape. Set max_tokens to a value close to what you actually need — it doesn't reduce quality but it prevents runaway verbosity and limits unexpected cost spikes.

Problem 4: Not Using Prompt Caching

If your system prompt, a large document, or a long conversation history repeats across requests, you're paying full price for it every time — unless you enable prompt caching.

Anthropic's prompt caching charges 1.25x on the first write (5-minute TTL) or 2x for a 1-hour TTL, then only 0.10x (10% of normal input price) on cache hits. If your system prompt is 2,000 tokens and you make 1,000 calls per day, without caching you're paying 2,000 × 1,000 × $3/MTok = $6/day just for the system prompt. With caching, you pay $0.0075 for the write and $0.60 for 999 cache reads — $6.00/day becomes $0.61/day, a 10x reduction on that portion of the cost.

See our full breakdown: Does prompt caching actually save money?

Fix

Add "cache_control": {"type": "ephemeral"} to stable content blocks in your API calls. On Anthropic, the minimum cacheable block is 1,024 tokens. Put your cache breakpoint after the system prompt and any static documents. For conversations, cache at the end of the conversation history so only the latest exchange is billed at full price.

Problem 5: Wrong Model for the Task

This is the most common and most expensive mistake. Teams default to a frontier model (GPT-4o, Claude Sonnet, Gemini 2.5 Pro) for everything because it "just works", then never revisit the choice as the application scales.

The cost difference between model tiers is enormous. Claude Haiku 4.5 ($1/$5 per MTok) is 3x cheaper on input and 3x cheaper on output than Claude Sonnet 4.6 ($3/$15). For tasks that don't require frontier intelligence — intent classification, simple Q&A, format conversion, data extraction from structured inputs — Haiku performs comparably and costs a fraction.

Fix

Implement model routing based on task complexity. Route simple, high-volume tasks (classification, extraction, summarization) to a cheap model. Route complex, low-volume tasks (reasoning, code review, writing) to a frontier model. A well-tuned two-tier router typically reduces overall costs by 50–70% with minimal quality impact. Test quality on a sample before committing to the switch.

Problem 6: No Budget Enforcement

The above five problems compound when there's no real-time cost tracking or hard spending cap. A small regression — an extra tool added to the system prompt, a loop that runs one extra turn, a context window that doesn't get trimmed — can silently double your costs over a week without triggering any alert.

Provider dashboards show you monthly totals. They don't show you which endpoint, which user, or which code change caused the spike. By the time the bill arrives, it's too late.

A real scenario: the agent retry loop

An agent calling an external API gets a rate limit error. The retry logic retries 5 times with the full conversation context on each attempt. Each failed call costs the same as a successful one. If this happens 100 times per day and your context is 5,000 tokens, you're spending $2.25/day ($67/month) on API calls that produce nothing — and you won't see it unless you're tracking per-call costs.

The Compound Effect

These six problems don't add up linearly — they multiply. A team running a frontier model (5x cheaper alternative available), with verbose outputs (10x shorter possible), growing context windows (4x smaller with summarization), and no prompt caching (10x cheaper with caching) could theoretically be paying 2,000x more than necessary for the same functionality.

In practice, a realistic team with just three of these problems unaddressed is typically paying 10–20x what they need to. The fixes aren't hard — they just require visibility into what's actually being sent and received on every call.
