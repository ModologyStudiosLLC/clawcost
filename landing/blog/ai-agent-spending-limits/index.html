How to Set Hard Spending Limits on Your AI Agents Before They Run Up a $500 Bill

AI agents are autonomous by design. That's the point. But without cost guardrails, a single stuck agent, a bad retry loop, or an unexpected multi-agent spawn can consume your entire monthly budget before you notice.

Unlike a traditional API call — where you send a request and get a response — an AI agent can make dozens or hundreds of LLM calls autonomously to complete a task. Each call costs money. A task that should take 5 calls can take 50 if the agent gets confused, loops, or decides to spawn sub-agents to help.

There is no built-in rate limiter at the LLM provider level that protects you from this. Anthropic, OpenAI, and Google will happily process every call your agent sends and bill you for all of them. The responsibility for spending control falls entirely on you.

Why Standard Per-Call Limits Don't Work

Most developers set a max_tokens limit and call it done. This limits the output size of each individual call — it does not limit how many calls an agent makes or the total money spent.

Provider accounts have monthly spend limits, but these are high (often $100–$500+ by default for new accounts) and aren't configurable per-agent, per-user, or per-task. By the time you hit the account limit, you've already overspent massively.

A pattern that causes $500+ incidents

An agent is given a research task. It decides to call a web search tool to get more context. The tool returns an error. The agent retries. The error persists (rate limit on the search API). The agent tries a different approach, calls the LLM again to reformulate the query, then retries the search again. This loop runs overnight. Each LLM call passes the full growing conversation history. You wake up to a $400 bill for a research task that never completed.

How to Calculate a Per-Task Budget

Before you can set limits, you need a realistic cost model for your agent. Here's the framework:

Agent Task Cost Estimator
Expected turns per taskN turns
Avg input tokens per turn (context + tools + prompt)T_in tokens
Avg output tokens per turn (reasoning + tool calls)T_out tokens
Input costN × T_in × (input_price / 1M)
Output costN × T_out × (output_price / 1M)
Total expected costInput cost + Output cost

Example: Customer Support Agent on Claude Sonnet 4.6

  • Expected turns: 6 (average support conversation)
  • Input per turn: 3,000 tokens (system prompt 1,200 + history + user message)
  • Output per turn: 400 tokens (response + tool calls)
  • Input cost: 6 × 3,000 × ($3 / 1,000,000) = $0.054
  • Output cost: 6 × 400 × ($15 / 1,000,000) = $0.036
  • Total expected cost per conversation: $0.09

Your per-task budget should be 3–5x the expected cost to absorb variance without cutting off legitimate tasks. In this case: $0.27–$0.45 per task.

Your daily budget should be: daily task volume × per-task budget × safety factor. For 500 conversations/day: 500 × $0.45 × 1.5 = $337.50/day hard cap.

The Three Layers of Agent Cost Control

Layer 1: Turn Limits (in code)

The simplest protection: limit the number of tool call iterations the agent can take before terminating. This is agent-level logic, not LLM-level.

MAX_TURNS = 15

async def run_agent(task, model, tools):
    messages = [{"role": "user", "content": task}]
    turns = 0

    while turns < MAX_TURNS:
        response = await call_llm(model, messages, tools)
        turns += 1

        if response.stop_reason == "end_turn":
            return response.content

        # handle tool calls...
        messages.append(tool_result)

    raise AgentError(f"Task exceeded {MAX_TURNS} turns — aborting")

Set MAX_TURNS to 2–3x your expected turns. A support agent that normally takes 6 turns gets a limit of 15. If it's still running at turn 15, something has gone wrong and it's better to abort and alert than to keep billing.

Layer 2: Per-Task Token Budget

Track cumulative tokens across all turns of a single task. If total input + output tokens exceed your budget, abort the task.

TOKEN_BUDGET = 50_000  # ~$0.45 at Sonnet 4.6 rates

class AgentSession:
    def __init__(self):
        self.total_input_tokens = 0
        self.total_output_tokens = 0

    def track(self, response):
        self.total_input_tokens += response.usage.input_tokens
        self.total_output_tokens += response.usage.output_tokens

    @property
    def estimated_cost(self):
        return (
            self.total_input_tokens * 3 / 1_000_000 +
            self.total_output_tokens * 15 / 1_000_000
        )

    def check_budget(self):
        if self.estimated_cost > 0.45:
            raise BudgetExceeded(f"Task spent ${self.estimated_cost:.3f}")

Layer 3: Global Daily/Monthly Hard Cap via Proxy

Application-level limits are good, but they have a gap: they only protect you from the agents you wrote correctly. A bug that bypasses your limits, an untested code path, a newly deployed agent with misconfigured settings — none of these are caught by code you wrote in the agent loop.

The only way to enforce a truly hard limit is at the network layer, between your code and the LLM provider API. A cost proxy intercepts every API call, tracks cumulative spend, and returns an error (instead of forwarding the request) once the daily or monthly cap is reached.

What to Do When a Limit Is Hit

A hard limit shouldn't just silently kill the agent. Your application needs to handle the budget exceeded state gracefully:

  1. Return a safe error to the user. "I wasn't able to complete this task. Please try a simpler version of your request." is better than a crash.
  2. Log the partial work. The agent may have completed 80% of the task before hitting the limit. Save intermediate results so the work isn't entirely lost.
  3. Alert your team. A budget hit is a signal that either the task was genuinely complex, or something went wrong. A Slack or email alert via webhook is easy to add to the limit handler.
  4. Review the conversation. Check what the agent was doing before hitting the limit. Loops and confused states are usually obvious in the logs.
